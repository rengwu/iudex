# iudex — Product Requirements Document

Status: ready-for-agent

---

## Problem Statement

Engineering teams that want to use AI coding agents (Claude Code, Aider, etc.) at scale face a coordination problem: agents need isolated workspaces, parallel work creates merge chaos, and human oversight must be inserted at the right points without becoming a bottleneck. Existing solutions either run agents sequentially (wasting throughput), require custom infrastructure (daemons, databases), or abandon human-in-the-loop review entirely. The result is that most teams run a single agent at a time and manually stitch together its output — a workflow that doesn't scale.

---

## Solution

iudex is a Go CLI tool that orchestrates multiple AI coding agents in parallel across git worktrees. It imposes a structured pipeline — queue → implement → QA → human review → merge — where each stage is file-based and git-native. Every ticket gets its own worktree; every state transition is recorded in an append-only event log; no agent can merge to main without explicit human approval. The human operator sees a live TUI showing active agents, spawn commands to copy-paste, stall alerts, and work awaiting review.

---

## User Stories

### Setup and initialisation

1. As a developer, I want to run a single init command in any directory, so that a complete iudex workspace is scaffolded without manual directory or git setup.
2. As a developer, I want iudex to initialise a git repository for me if one doesn't already exist, so that I can start orchestrating from a blank project.
3. As a developer, I want to edit a single config file to set agent concurrency, poll interval, stall timeout, and agent command, so that I can tune orchestration behaviour without touching code.
4. As a developer, I want the impl and QA system prompts to be editable files on disk rather than hard-coded strings, so that I can customise agent behaviour per project.
5. As a developer, I want the binary to embed all templates so that initialising a workspace requires no network access and no extra files beyond the binary itself.
6. As a developer, I want iudex to detect an existing git repository on init and skip re-initialisation, so that I can add iudex to an existing project safely.

### Ticket creation and queue management

7. As a developer, I want to create a ticket with a single command that accepts an ID, a title, and optional dependency IDs, so that I can describe work items without a GUI.
8. As a developer, I want to pre-write a ticket file in `queue/` and have iudex register it automatically when I run `new-ticket`, so that I can write rich briefs in my editor before involving iudex.
9. As a developer, I want to set a numeric priority (1–5) on a ticket, so that the orchestrator can claim higher-priority work first when multiple tickets are queued.
10. As a developer, I want to declare dependencies between tickets using `--deps`, so that a dependent ticket is never started until all its blocking tickets are in `done` state.
11. As a developer, I want `--deps` to inject a `## Dependencies` section into a pre-existing ticket file if one is absent, so that dependency enforcement works regardless of whether the ticket was pre-written or created fresh.
12. As a developer, I want the queue to hold as many tickets as I like without affecting active agents, so that I can plan work ahead without triggering premature claiming.
13. As a developer, I want to see the full list of ticket states at any time without starting the TUI, so that I can check pipeline status from scripts or SSH sessions.

### Agent orchestration

14. As a developer, I want the orchestrator to claim queued tickets automatically up to a configured concurrency limit, so that I don't have to manually start agents on individual worktrees.
15. As a developer, I want the orchestrator to skip tickets whose declared dependencies are not yet `done`, so that agents never start work on a ticket before its prerequisites are complete.
16. As a developer, I want the TUI to surface a `⏳ ticket-NNNNN waiting on: ticket-XXXXX (state)` alert whenever a ticket is blocked by an unmet dependency, so that I can immediately see why a ticket is not being claimed.
17. As a developer, I want the orchestrator to surface a ready-to-paste spawn command per active ticket showing the exact `cd` path and agent invocation, so that I can open an agent session with a single copy-paste.
18. As a developer, I want the spawn command to distinguish between `impl` and `qa` roles with colour-coded badges, so that I can tell at a glance which type of agent to launch.
19. As a developer, I want the orchestrator to detect stalled agents (no commits for N minutes) and alert me, so that I notice hung or crashed agents without having to monitor worktrees manually.
20. As a developer, I want the orchestrator to auto-commit any uncommitted changes in a worktree before handing it off to a QA agent, so that the QA diff is always complete.
21. As a developer, I want the orchestrator to surface a QA spawn command automatically when a ticket enters `pending-review`, so that QA review starts without manual intervention.
22. As a developer, I want the orchestrator to re-derive spawn commands on every tick, so that tickets returning to `in-progress` via `iudex revise` are covered without restarting the TUI.

### Impl agent workflow

23. As an impl agent, I want a pre-populated `.task/brief.md` in my worktree describing the ticket, so that I don't have to look elsewhere for requirements.
24. As an impl agent, I want a `.task/log.md` file ready for notes, decisions, and handoff instructions, so that context is preserved across agent sessions.
25. As an impl agent, I want `.task/` to be gitignored so that my working notes never land in the commit history.
26. As an impl agent, I want to signal completion by running `iudex finish <id>` from anywhere in the workspace, so that the ticket transitions to `pending-review` without me touching git directly.
27. As an impl agent, I want any uncommitted changes auto-committed when I call `iudex finish`, so that no work is lost during handoff.

### QA agent workflow

28. As a QA agent, I want to read `.task/brief.md`, `.task/log.md`, and the diff against `main` in one place, so that I can assess the implementation without hunting for context.
29. As a QA agent, I want to write a structured review to `.task/review.md`, so that my findings are available to the human reviewer.
30. As a QA agent, I want to approve work by running `iudex finish <id>`, so that the ticket moves to `pending-human-review`.
31. As a QA agent, I want to request revision by running `iudex revise <id>`, so that the ticket returns to `in-progress` and my review is preserved for the impl agent.
32. As a QA agent, I want to know that my role is read-only relative to the implementation, so that I never accidentally modify production code while reviewing.

### Human review and merge

33. As a human reviewer, I want `iudex review <id>` to print the brief, implementation log, diff against main, and QA review in sequence, so that I can assess a ticket in one terminal session.
34. As a human reviewer, I want `iudex review` to print the exact merge and reject commands I need next, so that I never have to recall command syntax under pressure.
35. As a human reviewer, I want `iudex merge <id>` to refuse unless the ticket is in `pending-human-review` or `human-manual`, so that incomplete work can never reach main.
36. As a human reviewer, I want merge to use `--no-ff` so that the branch history is preserved on main alongside a clear merge commit.
37. As a human reviewer, I want the merge commit message to follow a consistent format (`feat: complete <ticket-id>`), so that the git log is scannable without reading individual commits.
38. As a human reviewer, I want `iudex reject <id> --reason "..."` to archive the ticket and return its brief to the queue, so that rejected work can be retried without losing the original requirements.
39. As a human reviewer, I want `iudex manual <id>` to let me take over a worktree for hands-on fixes, so that I'm not blocked when an agent produces code I need to adjust myself.
40. As a human reviewer, I want `iudex finish <id>` after a manual session to hand the ticket back to QA, so that manual work goes through the same review gate as agent work.
41. As a human reviewer, I want every merge and rejection decision to be explicit and irreversible without manual git intervention, so that I'm always aware of what reaches main.

### Archival and history

42. As a developer, I want every completed or rejected ticket archived with its brief, log, QA review, diff patch, and a metadata JSON file, so that I have a permanent record of what was built and why.
43. As a developer, I want the archive directory to record the merge commit hash, outcome, timestamp, and full event history for each ticket, so that I can reconstruct the decision trail without reading git log.
44. As a developer, I want rejected archives to live under a `_rejected` suffix so they are visually distinct from completed ones.
45. As a developer, I want `iudex archive-list` to show at a glance which archived tickets have diffs and QA reviews, so that I can quickly find tickets with full records.

### TUI and monitoring

46. As a developer, I want a live terminal dashboard that shows spawn commands, queue depth, active agents, tickets awaiting review, and alerts in labelled panels, so that I can monitor the full pipeline at a glance.
47. As a developer, I want the TUI to refresh automatically on every orchestrator tick and on demand with `r`, so that the display stays current without me polling manually.
48. As a developer, I want stall alerts to be dismissible as a group with `a`, so that I can acknowledge noise without quitting the TUI.
49. As a developer, I want the TUI to show the commit count and time-since-last-commit per active ticket, so that I can judge agent progress without leaving the dashboard.
50. As a developer, I want to quit the TUI with `q` or `ctrl+c` without killing active agent processes, so that agents continue working in their own terminal sessions.

### Dependency enforcement

51. As a developer, I want the orchestrator to block a ticket from being claimed if any declared dependency is not in `done` state, so that the work ordering I specified is always respected.
52. As a developer, I want a missing ticket file (TOCTOU race between glob and deps check) to be treated as an error rather than "no deps," so that a disappearing queue file can never bypass dependency enforcement.
53. As a developer, I want blocked tickets to show in the TUI ALERTS panel with the exact blocking dep IDs and their current states, so that I can diagnose ordering issues without reading the event log.

## Implementation Decisions

- **State machine via append-only JSONL.** All ticket state lives in `events.jsonl`. A single replay over that file produces the current state of every ticket. No database, no locking, no runtime daemon. Concurrent CLI invocations are safe because the kernel guarantees `O_APPEND` atomicity on POSIX filesystems.

- **One git worktree per active ticket.** Each ticket gets `project/worktrees/<ticket-id>/` branched from `main`. The impl and QA agents work in this directory; the human always merges from it. Worktrees are created on claim and removed on merge or rejection. The branch name is always `work/<ticket-id>`.

- **`.task/` directory inside the worktree.** Brief, log, and QA review live inside the worktree under `.task/`, gitignored. This colocalises context with the work without polluting the commit history or requiring path arithmetic across worktrees.

- **Dependency declarations in the ticket file.** A `## Dependencies` section with `- <ticket-id>` lines is the sole source of dependency data. The orchestrator reads this from the queue file on every tick and compares against the current state map from `events.jsonl`. No separate dependency store is needed.

- **`ParseDependencies` returns an error for missing files.** A file absent at deps-check time (TOCTOU) is treated as an error, which causes the orchestrator to alert and skip rather than treat the ticket as dep-free. This closes a safety gap where "file gone" would previously be read as "no deps → ready to claim."

- **Orchestrator surfaces commands; humans launch agents.** The orchestrator never `exec`s agent processes. It computes the correct spawn command (including the system-prompt argument) and surfaces it to the TUI. The human copy-pastes it into a terminal. This keeps iudex agent-agnostic — any tool that accepts a prompt argument works.

- **Merge is always `--no-ff`.** Squash merge was considered but rejected in favour of `--no-ff` to preserve per-ticket branch history on main. Each merge produces one merge commit with message `feat: complete <ticket-id>`.

- **Dep-blocked alerts rebuilt every tick.** Rather than accumulating and pruning dep-blocked alerts, the orchestrator recomputes them from scratch on each tick and replaces any `⏳`-prefixed alert with the fresh set. This avoids stale alerts when deps become satisfied between ticks.

- **`--deps` applied to pre-existing queue files.** When `iudex new-ticket` is called on a file that already exists but lacks a `## Dependencies` section, and `--deps` is non-empty, the command injects the section before `## Problem Statement`. An existing `## Dependencies` section is never overwritten — the operator edits it manually if they want to change declared deps.

- **All git operations via `exec.Command`.** No libgit2. Works on any system where `git` is in `$PATH`.

- **Templates embedded in the binary.** The `//go:embed all:templates` directive bundles the `.iudex/` scaffold (config, system prompts, skills) inside the binary. `iudex init` extracts them; no network access or separate install step is required.

## Testing Decisions

A good test asserts on observable outcomes through the module's external interface — what comes out of the system given what goes in — without inspecting internal state or mocking away the thing under test. Tests at a higher seam are preferred; a failing test at the CLI seam is more valuable than ten passing unit tests on helpers.

**Three seams, high to low:**

### CLI binary seam (highest)
End-to-end tests compile the binary and invoke it via `exec.Command` against a real git repository in a temporary directory. Each test creates a workspace with `iudex init`, drives the pipeline through `iudex new-ticket`, `iudex finish`, `iudex merge`, etc., and asserts on exit codes, stdout content, the final state in `events.jsonl`, and the presence or absence of worktrees and archive files. This seam catches integration failures that no unit test can — git ops, event log serialisation, worktree lifecycle, and archive correctness all exercised together.

### events.jsonl seam
Tests call `events.Append` and `events.GetAllTickets` directly against a temp directory. They verify state replay (does replaying the log produce the expected current state?), concurrent append safety (multiple goroutines writing simultaneously should not corrupt the log), and edge cases (empty file, malformed lines silently skipped). Prior art: the existing `replayStates` function structure is already well-suited to table-driven tests.

### queue/deps seam
Tests call `ParseDependencies`, `DepsReady`, and `BlockingDeps` against fixture `.md` files written in the test. They verify that the `## Dependencies` section is found regardless of its position in the file, that missing files return an error (not nil), that dep IDs are parsed exactly (no trailing whitespace), and that `DepsReady` returns false for any dep not in `done` state. No git or filesystem setup beyond temp files needed.

## Out of Scope

- **Automatic merging.** Human approval via `iudex merge` is always required. No auto-merge on QA pass.
- **Remote or multi-machine coordination.** All state is local. No network sync, no shared event log across machines.
- **Web dashboard.** The TUI is the only monitoring surface in v1.
- **AI-driven ticket assignment or bundling.** The orchestrator claims tickets in queue order (subject to priority and deps); it does not analyse tickets to group or assign them.
- **`feedback` command.** Referenced in earlier design notes but not yet implemented; out of scope for this PRD.
- **Integration tests against real git repos for all git functions.** The git package functions are implemented but lack end-to-end test coverage; this is a known gap tracked separately.
- **Automatic retry on agent failure.** If an agent crashes, the ticket stalls and alerts; a human must intervene via `iudex manual` or `iudex reject`.
- **Cross-platform builds beyond Linux/macOS and Linux/ARM.** The Raspberry Pi 3B cross-compile target is supported but other targets are not tested.

## Further Notes

- iudex is a Go rewrite of a Python prototype. The workspace layout, pipeline states, and file conventions are locked in and should not change without a deliberate ADR.
- The CLAUDE.md in the repo root doubles as the agent handoff document. It is the authoritative source of design decisions and should be kept in sync with any implementation changes.
- The seven valid ticket states are: `queued`, `in-progress`, `pending-review`, `pending-human-review`, `human-manual`, `done`, `rejected`. No other states should be introduced without updating the event log schema and all consumers.
- The orchestrator goroutine is the only writer to `events.jsonl` for state transitions it controls (claim → `in-progress`). CLI commands (`finish`, `merge`, `reject`, `revise`, `manual`) also write transitions. Concurrent writes are safe because `O_APPEND|O_WRONLY` is used; the replay model tolerates interleaving.
